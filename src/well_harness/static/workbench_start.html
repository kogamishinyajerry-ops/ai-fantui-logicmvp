这页在按"角色 + 工作意图"两个维度组织入口

  • P1–P5 代表常见工程工作方式：学习走读、资深调试、汇报演示、 适航追溯和客户问题复现。每张卡片都指向主面板里的当前可用工作流。
  • KOGAMI 不是 persona，是项目内部的审批权限角色（仅 Kogami 可签发 proposal）。 与 P1–P5 同列展示，是因为审批入口和上面五种工作一样来自 Workbench。
学习与演示

跑 4 张一键验收预设

P1 · Junior FCS · 30 分钟内上手

落到 /workbench「一键预设验收卡」。 当前 4 张： 通过并留档 / 阻塞演示 / 快速通过 / 留档复跑，每张点完会自动跑 intake → bundle 真实链路；其中 ready_archived 与 archive_retry 还会 落 archive 目录，blocked_follow_up 与 ready_preview 不落 archive。 不要求先碰 JSON。

  • 4 张预设：ready_archived / blocked_follow_up / ready_preview / archive_retry
  • archiveBundle: true 仅 2/4（ready_archived + archive_retry）
  • 当前主面板顶部是控制逻辑电路；场景演示入口在左侧视图和 /demo.html
→ 工程师调试

读 3 列 shell + Raw JSON 抽屉

P2 · Senior FCS · 真值只读 / 提案受签

落到 /workbench 的 control / document / circuit 三列 shell + 「开发调试 / Raw JSON」 抽屉。 改动 spec 不会改真值，只能转成 ticket / proposal 走签批。 TRA / RA / N1k 等 lever 调参在 /demo.html；主面板聚焦电路、证据、提案和归档。

  • truth-engine 是 read-only（红线见底部）
  • 顶部 ticket / identity chrome 默认显示，可读不可改
  • L1–L4 逻辑电路和当前现状条在主面板可见，均为只读/参考面
→ 立项 / 汇报演示

用「通过并留档」预设走 happy path

P3 · Demo Presenter · 上台前最后过一遍

进入主面板后，先用 「一键通过验收」（ready_archived） 走通 happy path， 再用 「一键看阻塞态」（blocked_follow_up） 展示 clarification gate。 需要 TRA / RA / N1k lever-driven 现场演示时，使用 /demo.html。

  • 建议演示前隐藏浏览器侧栏 + 关闭 Approval Center 抽屉
  • "一键留档复跑" 适合演示同秒重复点也不崩
  • 主面板保留完整工程证据；演示舱用于讲故事，不替代验证证据
→ 客户问题复现

填 9 个 knowledge 字段

P5 · Apps Engineer · intake → bundle → archive

落到 /workbench 的 knowledge 区（"Resolution / Knowledge 字段"折叠面板）。 当前 schema 是 9 字段： Observed Symptoms / Evidence Links / Confirmed Root Cause / Repair Action / Validation After Fix / Residual Risk / Suggested Logic Change / Reliability Gain Hypothesis / Guardrail Note。 客户邮件原文先落到 Evidence Links，再进入 intake → bundle → archive 链路。

  • knowledge 字段已经在主面板渲染（dt/dd 列表）
  • 复现步骤、截图引用和客户原话应保留在证据字段里
  • 提交后转到 intake → bundle 真实链路，不新增第二套规则
→ 提案审核（Kogami）

Approval Center · 提案入口

Kogami-only · 签批必须保留证据链

落到 /workbench 底栏的 Approval Center 入口。这里用于审阅 proposal 状态和签批边界；真实代码真相仍在 GitHub/repo，工作控制真相仍在 Linear。 浏览器入口不能绕过证据链，也不能直接改控制真值。

  • 审批动作必须能追溯到 ticket、proposal 和 repo 证据
  • 非 Kogami 角色只能读状态，不能签发 proposal
  • 任何合并或关闭动作都以 GitHub/Linear 证据为准
→ V&V 适航追溯

沿 intake → bundle → archive 走读

P4 · QA / V&V · DO-178C 风格审计

从某个 acceptance preset 跑出 bundle，去 archive 包里读结果， 对照 controller.py + adapters/*.py 找 requirement 引用。 顶部当前现状条只给审计方向，不等于现场测试或适航结论。

  • archive package 自带 timestamp + slug 目录 + SHA256 文件完整性哈希（不是 git commit SHA）
  • truth-engine SHA / e2e / adversarial 状态条是参考证据，不是实时真值读数
  • 当前产出物均为内部研究证据，不构成适航合规声明
→

红线 · 你点哪一张都不会改的东西

  • controller.py · 19-node truth engine · 4 logic gates — read-only
  • adapters/*.py 真值出口 — read-only
  • 已签 audit event 的 hash chain — append-only
  • 参考场景与证据包 — 只读复用，不在浏览器里改真值

所有"修改"都走 proposal → ticket → Kogami 签批 流程； 详见 Approval Center 与 主入口 · 工作台总览。
