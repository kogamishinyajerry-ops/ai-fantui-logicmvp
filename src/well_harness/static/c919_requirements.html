§ 1

C919 E-TRAS 系统架构

C919（商用飞机有限责任公司）采用电动反推作动系统（E-TRAS），相比传统液压反推有以下根本差异：

传统液压反推： C919 E-TRAS（电动）： ┌─────────────┐ ┌──────────────────────────┐ │ 液压管路 │ │ 115VAC + 540VDC 电气总线 │ │ 机械开关 │ → │ EICU 双通道控制器 │ │ 单一路径 │ │ FADEC 接口（部署命令） │ └─────────────┘ │ TLS/PLS 电磁锁 │ │ PDU 电机 + 翻转斗 │ └──────────────────────────┘

EICU（电子接口控制组件）双通道冗余架构：EICU 有两个独立处理通道（CMD1 / CMD2），任意一个通道健康即可发出控制命令。这与 FANTUI 的单一逻辑链有根本区别——C919 需要表达"哪个通道激活、是否降级运行"这类有状态概念。

FADEC 接口：全权数字发动机控制器（FADEC）是反推展开的最终执行门——FADEC_Deploy_Command 和 FADEC_Stow_Command 由控制逻辑产生，FADEC 执行后才真正驱动 PDU 电机。这增加了一个间接层，需要在状态机中单独表达"FADEC 已收到命令"这一中间状态。

TLS + PLS 机械锁序列：C919 的锁机构比 FANTUI 简化模型更复杂——TLS（位移锁）负责解锁 TR 翻转斗的初始位移，PLS（主锁，共 4 路：左右各 2 个）在展开过程中逐步解锁。整个序列有严格的时序要求。

AI 架构选择依据

当 AI 分析 C919 E-TRAS 规范时，发现 FANTUI 的无状态架构无法直接适用的核心原因：锁机构的解锁状态无法从当前输入信号推断——TLS 是否已解锁取决于过去是否曾经供过电 0.3s 以上，这是历史状态，不是当前信号。

进一步分析发现，C919 规范文件本身就使用"状态"来描述系统行为（"在 S3 状态下，接收到 WOW 信号时转移到 S4"），这强烈提示状态机是最自然的实现模型。AI 据此决定：将 FANTUI 的纯函数逻辑扩展为有限状态机（FSM），每个状态对应一组稳定的输出命令集合。

§ 3

需求规范

架构需求（Architecture Requirements）

AR-01电气总线双路供电
系统必须区分 115VAC 单相供电（用于 TLS/PLS 解锁）和 540VDC 三相供电（用于 TRCU/PDU 驱动），两路总线独立控制，不得混用。
来源: C919 电气架构规范 §4.2 · 对应输出: SinglePhaseUnlockPower_On / ThreePhaseTRCUPower_On
AR-02EICU 双通道冗余
EICU 的两个控制通道（CMD1/CMD2）必须独立评估条件，其中任意一个通道健康且条件满足即可发出命令。双通道同时失效时进入 SF 故障状态。
来源: DO-178C DAL-A 要求的双通道冗余架构 · 面板中 LGCU1_WOW / LGCU2_WOW 分别对应两通道
AR-03FADEC 命令接口
控制逻辑不直接驱动 PDU 电机，而是向 FADEC 发出 FADEC_Deploy_Command 或 FADEC_Stow_Command，由 FADEC 负责最终执行。控制逻辑需等待 FADEC 确认信号。
来源: FADEC 接口控制文件 ICD-FADEC-TR-002

SM-01S0 → S1：地面装备
转移条件：Selected_MLG_WOW = True（主起落架接地信号，任意 LGCU 通道有效）AND engine_running = True。进入 S1 后，系统进入"地面装备"状态，等待飞行员拉反推杆。
SM-02S1 → S2：单相解锁供电
转移条件：SW1 激活（TRA 越过 -1.4° ~ -6.2° 区间）AND LGCU1/2 WOW AND engine_running。输出：SinglePhaseUnlockPower_On = True，开始向 TLS 和两侧吊挂锁供 115VAC。
SM-03S2 → S3：三相作动供电
转移条件：TLS_Unlocked = True AND PLS_L_Unlocked = True AND SW2 激活（TRA ≤ -5.0°）。输出：ThreePhaseTRCUPower_On = True，向 TRCU 供 540VDC。
SM-04S3 → S4：展开中
转移条件：FADEC_Deploy_Command 已发出 AND Unlock_Confirmed = True。进入 S4，FADEC 驱动 PDU 电机展开翻转斗，TR_Position 开始上升。
SM-05S4 → S5/S6：反推出力
TR_Position ≥ 90%（VDT90）时进入 S5（慢车反推）。飞行员继续拉杆至深反推区（TRA ≤ -28°）后进入 S6（最大反推）。
SM-06S6/S5 → S7：减速待收
转移条件：TRA 回到 0°（飞行员推杆）。系统进入减速等待收起状态，发出 FADEC_Stow_Command，准备驱动 PDU 反向运动。
SM-07S7→S8→S9→S10→S0：收起锁定
S8（收起中）：TR_Position 下降。S9（锁止确认）：TR_Position ≤ 0% AND TLS/PLS 重新锁定。S10（断电完成）：所有电气供电切断。S0（初始状态）：等待下次接地。
SM-SF任意状态 → SF：故障中止
故障触发条件：EICU 双通道同时失效 / 传感器完全失联 / 展开过程中检测到位置传感器异常 / 飞行员主动触发中止。SF 状态下：所有命令切断，需人工干预重置。
SF 是任意状态的全局转移目标，优先级最高，不被任何门所拦截

功能需求（Functional Requirements）

FR-C1翻转斗到位确认后才允许最大推力
与 FANTUI SR-04 等价：TR_Position < 90% 时不进入 S6（最大反推），且 TRA 深拉区（≤ -28°）在 S5 状态确认前不可用。
FR-C2锁机构完全到位才允许断电
进入 S10（断电完成）的条件之一是 TR_Stowed_And_Locked = True——确认翻转斗已物理归位且所有锁机构重新锁定，防止断电后机构漂移。
FR-C3过温保护
PDU 电机过温时（overtemp 信号），在 S4/S5/S6 状态下立即发出 FADEC_Stow_Command 并跳转 S7（收起序列），防止因长时间大电流导致电机损坏。
§ 4

AI 需求解析与状态机推导

AI 解析视角

AI 在处理 C919 E-TRAS 规范时，首先尝试将 FANTUI 的 4 门级联架构直接移植，但很快发现以下歧义无法在无状态框架下表达：(1) "TLS 已解锁"依赖历史；(2) "FADEC 已确认展开"是异步事件；(3) "过温时立即收起"是事件驱动转移，不是条件评估。因此 AI 切换到状态机模型，将每个稳定的系统行为点定义为一个状态。

推导路径：从 FANTUI AND 门到 C919 状态机

决策 C1
FANTUI L1/L2 的供电时序 → C919 S1/S2/S3 三级状态
FANTUI 中 L1 和 L2 在同一评估函数里，只要条件满足就同时为 True。但 C919 规范明确要求先完成 TLS 解锁（单相供电）才能开启三相供电。这意味着存在一个中间状态 S2，其中单相已上电但三相尚未激活。AI 将 L1 映射为 S2 的入口条件，L2 映射为 S3 的入口条件。
决策 C2
FADEC 的间接性 → 独立的 S3→S4 转移
FANTUI 的 L3 门直接产生 PDU 电机命令。C919 规范说"控制逻辑向 FADEC 发指令，FADEC 确认后才执行"。这个"等待确认"阶段不能在无状态函数中表达。AI 设计了 S3（三相供电已建立，等待 FADEC_Deploy_Command）→ S4（FADEC 已确认，展开中）的显式转移。
决策 C3
反推收起是独立路径 → S7~S10 专用收起链
FANTUI 的逻辑只处理展开方向（无收起路径，收起是植入模型自动处理）。C919 规范有详细的收起步骤：慢停 → FADEC 收起命令 → 等待 TR 归位 → 确认锁定 → 断电。AI 为此增加了 4 个独立的收起状态（S7/S8/S9/S10），与展开路径（S1~S6）对称。
决策 C4
故障处理 → SF 作为全局逃逸状态
FANTUI 通过"故障注入覆写节点"模拟故障，但这是测试工具行为，不是运行时行为。C919 规范要求系统本身在运行时检测故障并进入安全状态。AI 设计了 SF_ABORT_OR_FAULT 作为从任意状态均可转移的全局逃逸目标，优先级高于所有正常转移条件。

§ 6

仿真面板验证方法

在 C919 MFD 仿真面板（:9191） 中按以下步骤验证状态机需求。需先启动 C919 面板服务器：python3 scripts/c919_etras_panel_server.py

Step 1
验证 SM-01（S0→S1 接地装备）
点击「Step1: WOW接地」预设，观察状态从 S0 跳转到 S1（地面装备），状态机节点变为已激活色。面板标签页显示 1ph:OFF · 3ph:OFF，尚无供电。
验证需求: SM-01 + AR-02（WOW来自双通道）
Step 2
验证 SM-02（S1→S2 单相供电）
点击「Step2: SW1进入」（TRA=-2°），观察状态跳转 S2，日志行显示 1ph:ON。TLS 锁图标变为橙色（供电中），等待 0.3s 解锁。
验证需求: SM-02 + AR-01（单相独立）
Step 3
验证 SM-03（S2→S3 三相供电）
点击「Step3: SW2进入」（TRA=-7°，锁全解锁），观察跳转 S3，日志显示 1ph:ON · 3ph:ON。TRCU 功率总线建立，FADEC 命令仍为 OFF（等待 S4 触发）。
验证需求: SM-03 + AR-01（三相独立）
Step 4
验证 SM-04~05（S3→S4→S5 展开与到位）
点击「Step4: 展开指令」（TRA=-15°, TR=10%），观察 S4 激活，FADEC_Deploy=T。再点击「Step5: 80%展开」（TR=85%），状态仍在 S4。此时打开时序图，可见 DEPLOY 信号持续激活而 TR_pos 曲线上升。
验证需求: SM-04 + FR-C1（90%前不进S5）
Step 5
验证 SM-SF（故障中止）
在 S4/S5 状态下，点击「⚠ 抑制」或「🔥 过温」预设，观察状态跳转到 SF_ABORT_OR_FAULT（红色），所有命令立即切断，日志显示 fault 状态。这验证了 SF 的全局逃逸优先级。
验证需求: SM-SF + FR-C3（过温保护）
Step 6
验证完整收起链（S7→S10）
从 S6（Step6: 最大反推）依次点击 Step7（回杆）→ Step8（收起指令）→ Step9（回锁）→ Step10（断电），观察状态沿 S7→S8→S9→S10 顺序流转，最终回到 S0。打开时序图查看 STOW 命令曲线和 TR_pos 下降曲线。
验证需求: SM-06/07 + FR-C2（锁到位才断电）
