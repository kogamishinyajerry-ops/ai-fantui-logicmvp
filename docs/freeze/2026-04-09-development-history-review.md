# AI FANTUI LogicMVP 开发历史回顾

这是一份可以在 5 分钟左右讲完的、尽量通俗的项目开发回顾。

## 一、最开始我们先把“到底要做什么”说清楚了

这个项目一开始就不是“做一个很大的飞控系统”，也不是“随便做个能动的动画页面”。

我们真正要做的是一件更聚焦的事：

- 把已经确认过的反推 deploy 逻辑做成一个确定性的系统；
- 这个系统既能跑，又能解释；
- 最后还要能演示、能验证、能复盘。

这个需求确认很重要，因为它直接避免了两个常见跑偏：

- 跑偏成“看起来很厉害，但其实不可验证的 AI 玩具”；
- 跑偏成“想一步做到完整物理仿真，结果项目越来越重”。

所以项目一开始就立了一个最核心的原则：

- `controller.py` 是真值；
- 其他东西都只是围着这个真值服务。

这条原则后来帮我们挡掉了很多潜在混乱。

## 二、我们先搭了一个能被解释清楚的逻辑骨架

在做页面之前，我们先把底层骨架拆清楚了：

- `models.py` 负责稳定的数据结构；
- `controller.py` 负责 L1 到 L4 的控制逻辑；
- `switches.py` 负责 `SW1 / SW2` 的开关行为；
- `plant.py` 负责 `TLS / PLS / VDT` 这些反馈量；
- `runner.py` 负责把整条链路串起来跑。

这样做的好处很直接：

系统不是“黑盒子”。

我们始终可以回答这些问题：

- 当前输入是什么？
- 哪个开关亮了？
- 哪个逻辑门通过了？
- 输出命令是什么？
- 反馈量为什么变了？

也就是说，这个项目从一开始就不是靠“猜感觉”开发，而是靠可以追踪的状态开发。

## 三、然后我们把它从“调试工具”慢慢做成了“能演示的产品”

有了逻辑骨架之后，项目进入第二阶段：

不是只给工程师看，而是要给人演示。

于是我们逐步长出了两种界面：

- 一个是 CLI，适合看时间线、事件、解释、诊断；
- 一个是浏览器里的逻辑演示舱，适合现场说明和交互。

浏览器这一层并不是一次成型的，而是一点点长出来的。

我们先后加上了：

- 逻辑主板；
- 拉杆快照；
- 条件面板；
- presenter run card；
- 场景预设；
- 当前结论区；
- truth boundary 提示；
- Raw JSON 调试抽屉。

这里有一个一直没变的设计原则：

页面不是新的真值系统，它只是后端真值的展示层。

也就是说，页面可以帮你看懂逻辑，但不能自己偷偷发明一套新逻辑。

## 四、项目里花了很多时间做“看起来像小事，其实是关键体验”的修正

如果只看最终页面，很多人会觉得我们做的只是一些小修小补。

但实际上，项目里花了很大力气在这些非常关键的交互问题上：

- `TRA` 到底好不好拖；
- 左右方向和数值含义是不是一致；
- `-14°` 到底是不是一个“有条件放开的限值”；
- `VDT` 拖动后是不是立刻更新；
- 页面会不会太挤；
- 调 `VDT` 的时候能不能同时看到右边逻辑主板。

这些问题一旦没处理好，现场演示会非常难受。

我们中间确实踩过很多坑，比如：

- `TRA` 看起来像被错误锁死；
- `VDT` 控件虽然挪了位置，但没真正接回实时更新链路；
- 本地浏览器打开的其实是旧 server，所以看起来“代码改了但页面没变”；
- 页面排版一度被挤得太紧，影响同屏观察。

这些修正不是为了“好看一点”。
本质上是在修“产品行为”和“用户理解”之间的错位。

## 五、后来我们不只是在开发产品，也在开发“开发这件事本身”

项目做到后面，就不只是“把 demo 做出来”这么简单了。

我们还给项目本身加了一套开发中枢：

- Git + GitHub Actions；
- Notion 控制塔；
- GSD phase / plan / state 管理；
- 共享验证入口；
- QA 回写；
- UAT gap；
- 以及 Opus 4.6 的人工审查 gate。

这层能力的价值非常大。

它让我们不再只是“做完一版看看行不行”，而是变成：

- 代码真值在 repo / GitHub；
- 控制真值在 Notion；
- 自动验证持续运行；
- 主观判断只留给一个人工 gate。

也就是说，项目后期真正变强的，不只是功能，而是整个开发闭环。

## 六、到当前这一版，项目已经从“一个逻辑 harness”进化成“一个可演示、可验证、可回顾的系统”

如果回头看，整个项目其实走过了几个很清楚的阶段：

1. 先做 deploy 逻辑 harness；
2. 再补 CLI trace / explain / diagnose；
3. 再加确定性的 demo 问答层；
4. 再把它变成 presenter-friendly 的浏览器演示舱；
5. 再补 smoke、回归保护和交互硬化；
6. 最后再把“单点状态显示”扩展成“状态随时间变化的监控过程”。

当前最新的一步，其实很关键：

系统现在不只是告诉你“这一刻是什么状态”，
而是可以告诉你“这一段过程里，每个部件是怎么随时间变化的”。

比如在你定义的监测流程里，现在可以直接看到：

- `RA` 怎么从 `7.0 ft` 降下去；
- `TRA` 什么时候开始往 `-14°` 推；
- 为什么会卡在 `-14°`；
- `VDT` 怎么从 `0%` 增长到 `100%`；
- `SW1 / SW2 / TLS / ETRAC / EEC / PLS / L1-L4 / THR_LOCK` 又分别在什么时间点变化。

这让整个 demo 从“一个静态解释面板”，变成了“一个过程监控和讲解工具”。

## 技术亮点摘要

- **真值边界非常清楚**：`controller.py` 始终保持唯一控制真值，没有被前端偷偷复制一份。
- **架构是可解释的**：模型、开关、控制器、反馈、runner 各司其职，出了问题容易定位。
- **前端没有造假数据真相**：浏览器页面看到的关键状态都来自后端。
- **调试面很强**：CLI 时间线、事件、解释、诊断和 Raw JSON 让问题能被看见、被复现。
- **演示能力是逐步打磨出来的**：不是只有“能跑”，而是变成了“能讲、能切场景、能看边界、能同屏观察”。
- **交互细节被认真修过很多轮**：`TRA` 拖动语义、`VDT` 更新、锁位逻辑、页面密度都做过深度修正。
- **自动化开发闭环很完整**：GitHub Actions、共享验证、Notion 控制塔、GSD 状态管理已经串起来了。
- **人工审查边界很清楚**：自动化负责证据和执行，Opus 4.6 负责最后的主观收口。
- **新增了过程监控能力**：最新版本已经能把逻辑系统画成“随时间变化”的图，而不是只看一个瞬时状态。

## 一句话总结

这个项目是从一个小型的反推逻辑 harness，一步一步长成了一个确定性、可解释、可演示、可验证、还能被持续自动化管理的 demo 系统。
