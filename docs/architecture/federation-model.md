## 核心原则

> **整合不是合并，而是在同一底座上通过显式接口互通，跨域关联由人类工程师确认。**

这条原则是联邦架构的宪法级定义。任何架构决策若与这条原则冲突，必须返回这条原则进行裁决。

---

## 三层联邦架构

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: AI 解释层（概率性，跨域叙事）                   │
│  MiniMax 可以引用多个域的信息生成解释                      │
│  但不能创造跨域的因果关系                                 │
│  无注册关联的跨域推断 → 必须标注"未真值验证"              │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 跨域关联登记表（人工确认，确定性）              │
│  cross_domain_links.json                                 │
│  每条关联由人类工程师显式声明并审批                       │
├─────────────────────────────────────────────────────────┤
│  Layer 1: 独立 Adapter 联邦（完全隔离，确定性）           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │  TR      │ │  LG      │ │  BA      │ │  EFDS    │  │
│  │ adapter  │ │ adapter  │ │ adapter  │ │ adapter  │  │
│  │ +truth  │ │ +truth  │ │ +truth  │ │ +truth  │  │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘  │
└─────────────────────────────────────────────────────────┘
```

## 不变式

**每个 adapter 是一个独立的 truth 域（truth domain）。**

- 域内的 truth evaluation 是自洽的、可独立验证的
- 跨域的关联是显式声明的、人工确认的、可撤销的
- Adapter 之间无直接代码依赖

---

## 为什么不是单体

单体合并（将所有控制逻辑合并成一个 adapter）违反以下原则：

1. **航空工程现实**：真实飞机的控制系统是联邦的，每个系统有独立的 LRU（Line Replaceable Unit），通过定义明确的信号接口互通。架构应反映物理现实。

2. **可验证性**：单体 truth evaluation 无法单独验证某一子系统，测试组合爆炸。

3. **项目宪法**：`controller.py` 是 confirmed control truth，每个系统有自己的 controller，合并违反 adapter boundary 原则。

## 整合的三级门槛

| 层级 | 名称 | 条件 | 谁决定 | 在系统中的表现 |
|------|------|------|--------|--------------|
| **Level 0** | 共存 | 两个系统都托管在同一底座上 | 自动（onboard 即共存） | 各自独立的 adapter + canvas，系统切换器并列 |
| **Level 1** | 信号关联 | 域 A 的节点输出是域 B 节点的输入条件 | **人类工程师** 提供工程依据（spec/线路图/ATA章节号） | `cross_domain_links.json` 中一条记录；Canvas 显示跨域虚线；truth engine 自动查询关联节点状态 |
| **Level 2** | 联合推理 | 多域状态组合导致域外系统级结论 | **人类工程师 + Opus 4.6 架构审查** | 新建聚合 adapter，不拥有自己的 truth，聚合多个域的 truth 输出 |

---

## 禁止的整合行为

| 行为 | 原因 |
|------|------|
| AI 自动推断两个域有关联 | 概率模型不能创造确定性工程依赖 |
| 因"看起来相关"而合并两个 adapter | 没有信号级证据就不是整合，是猜测 |
| 在域 A 的 adapter 内硬编码域 B 的节点名 | 违反 adapter 隔离原则 |
| AI 的跨域解释被写入 truth evaluation | AI 是解释者不是决定者（P16 宪法原则） |

---

## 真相源的三层定义

1. **工程文档真相**：OEM 发布的控制逻辑图、线路图、ATA 手册中明确记载的信号依赖
2. **Adapter truth engine**：每个域内部的确定性推理引擎，其输出是该域的 ground truth
3. **人类工程师判断**：当工程文档不完整或存在歧义时，基于经验的关联声明——需记录声明人、日期，可被后续审查撤销

## 跨域推理流程

```
用户问："如果液压系统失效，起落架和反推力都会受影响吗？"
          │
          ▼
【Step 1】 Intent Router 识别：跨域查询
          涉及域：landing-gear + thrust-reverser
          查询 cross_domain_links.json
          │
          ▼
【Step 2】 检查是否存在已注册的跨域关联
          │
          ├─ 存在 Level 1 关联（如 hydraulic → LG 且 hydraulic → TR）
          │   ├─ 调用 LG adapter snapshot（with hydraulic=failed）
          │   ├─ 调用 TR adapter snapshot（with hydraulic=failed）
          │   ├─ 两个 Canvas 都更新
          │   └─ AI 基于两个 truth 结果生成跨域解释 ✅
          │
          └─ 不存在已注册的跨域关联
              ├─ AI 回答："根据航空工程常识，液压失效通常会影响起落架
              │   和反推力，但当前系统中没有注册这两个域之间的正式关联。"
              ├─ Canvas 不联动（无 truth 依据）
              └─ AI 回复中标注："⚠️ 以下是基于一般知识的推测，
                  尚未经过本系统的真值引擎验证" ✅
```

## 风险与缓解

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| 跨域 truth 循环依赖（A→B→A） | 🔴 | schema 强制关联单向；检测到循环则拒绝注册 |
| 跨域 truth 时间窗口不一致 | 🟠 | 跨域 snapshot 使用原子性聚合调用（一次 request 获取所有相关域） |
| 人工确认的关联过期 | 🟠 | spec 更新时自动标记相关关联为 `needs_review` |
| AI 跨域幻觉 | 🟡 | 无注册关联的推断必须标注"未真值引擎验证" |

## 宪法增补

> **跨域整合原则**：每个控制系统通过独立 adapter 接入，域内 truth evaluation 自洽且可独立验证。跨域关联只能由人类工程师基于工程文档显式声明并登记在 `cross_domain_links.json` 中。AI 不能创造跨域因果关系。在没有注册关联的情况下，AI 的跨域推断必须标注为"未经真值引擎验证的参考信息"。
