# P10-01: Landing-Gear Full Pipeline Proof -- Intake --> Playback --> Diagnosis --> Knowledge

## Context

P8-06 produced `two_system_runtime_comparison.py` which already runs the landing-gear adapter through the full playback --> diagnosis --> knowledge chain. However, that chain starts from `GenericControllerTruthAdapter.load_spec()` -- not from `ControlSystemIntakePacket`.

The generalized contract layer supports two entry points per stage:
- Adapter path: `build_*_from_truth_adapter()` -- takes a `GenericControllerTruthAdapter`
- Intake path: `build_*_from_intake_packet()` -- takes a `ControlSystemIntakePacket`

Landing-gear currently only exercises the adapter path. P10-01 needs to prove the intake path also works end-to-end, with each stage's JSON output validated against its v1 schema.

### What exists today

| Stage | Adapter path | Intake path | Schema file | Per-stage validation script |
|---|---|---|---|---|
| Intake (packet) | N/A | `intake_packet_from_dict()` | N/A (packet is not schema-validated; spec is) | None |
| Spec build | `load_spec()` | `intake_packet_to_workbench_spec()` | `control_system_spec_v1.schema.json` | `validate_landing_gear_adapter.py` (tests adapter.load_spec()) |
| Playback | `build_playback_report_from_truth_adapter()` | `build_playback_report_from_intake_packet()` | `playback_trace_v1.schema.json` | `validate_landing_gear_playback.py` |
| Diagnosis | `build_fault_diagnosis_report_from_truth_adapter()` | `build_fault_diagnosis_report_from_intake_packet()` | `fault_diagnosis_v1.schema.json` | `validate_landing_gear_diagnosis.py` |
| Knowledge | `build_knowledge_artifact_from_truth_adapter()` | `build_knowledge_artifact()` | `knowledge_artifact_v1.schema.json` | `validate_landing_gear_knowledge.py` |

### What's missing

1. **No end-to-end intake path validation**: No single script runs `ControlSystemIntakePacket` from landing-gear through all four stages and validates each output against its v1 schema.
2. **No landing-gear intake packet JSON artifact**: Landing-gear has no `document_intake.py` entry point -- its `load_spec()` bypasses `ControlSystemIntakePacket`. Need to build a landing-gear intake packet from the adapter's spec.
3. **23 shared validation commands reference**: The exit criteria mention "23 shared validation commands". These are the combined tests from the 4 existing per-stage validation scripts plus the `two_system_runtime_comparison` validator (4 + 4 + 4 + 4 + 1 + 2 smoke tests + 4 smoke schema tests = 23). All must pass with no regression.

## Tasks

### Task 1: Build landing-gear `ControlSystemIntakePacket` from adapter spec

Create `src/well_harness/adapters/landing_gear_intake_packet.py` with a `build_landing_gear_intake_packet()` function that converts the landing-gear `ControlSystemWorkbenchSpec` dict into a `ControlSystemIntakePacket` (the intake-layer schema).

This is the missing bridge: without an intake packet, the intake path of the pipeline cannot be exercised.

**Key design decisions:**
- The landing-gear adapter's `build_landing_gear_workbench_spec()` already produces a complete spec with components, logic nodes, acceptance scenarios, and fault modes.
- The intake packet needs additional fields that the adapter spec doesn't have: `source_documents` (can use a stub reference to `landing_gear_adapter.py`), `clarification_answers` (can be empty -- defaults are acceptable).
- Reuse `document_intake.intake_packet_from_dict()` to convert the dict back to a typed `ControlSystemIntakePacket`.

### Task 2: Build the end-to-end intake-path validation script

Create `tools/validate_landing_gear_full_pipeline.py` that:
1. Builds the landing-gear `ControlSystemIntakePacket` via Task 1.
2. Runs `intake_packet_to_workbench_spec()` and validates the resulting spec dict against `control_system_spec_v1.schema.json`.
3. Runs `build_playback_report_from_intake_packet()` and validates the playback report dict against `playback_trace_v1.schema.json`.
4. Runs `build_fault_diagnosis_report_from_intake_packet()` and validates the diagnosis report dict against `fault_diagnosis_v1.schema.json`.
5. Runs `build_knowledge_artifact()` and validates the knowledge artifact dict against `knowledge_artifact_v1.schema.json`.
6. Prints a machine-readable summary (JSON or text).

Each stage's schema validation must pass independently. This proves the generalized contract layer correctly hosts the landing-gear adapter through every pipeline stage.

### Task 3: Add pytest coverage for the full pipeline validation script

Add `tests/test_landing_gear_full_pipeline_validation.py` that:
- Calls `tools/validate_landing_gear_full_pipeline.py` as a subprocess (smoke + JSON output modes).
- Verifies the exit code is 0 and all stage schemas pass.
- Verifies graceful skip when `jsonschema` is not installed.
- Covers both `--format text` and `--format json` output modes.

### Task 4: Verify the 23 shared validation commands still pass

The "23 shared validation commands" are the aggregate of:
- `validate_landing_gear_adapter.py`: 4 cases (`landing_gear_adapter_metadata`, `landing_gear_control_system_spec`, `landing_gear_nominal_runtime`, `landing_gear_blocked_runtime`)
- `validate_landing_gear_playback.py`: 2 cases (`landing_gear_adapter_backed_playback_schema`, `landing_gear_adapter_backed_playback_alignment`)
- `validate_landing_gear_diagnosis.py`: 2 cases (`landing_gear_adapter_backed_diagnosis_schema`, `landing_gear_adapter_backed_diagnosis_contract`)
- `validate_landing_gear_knowledge.py`: 2 cases (`landing_gear_adapter_backed_knowledge_schema`, `landing_gear_adapter_backed_knowledge_contract`)
- `validate_second_system_smoke_schema.py`: 1 script (smoke)
- `test_second_system_smoke.py` + `test_second_system_smoke_schema.py`: 4 test methods each
- `two_system_runtime_comparison.py`: exercised via `test_second_system_smoke.py` (implicit)

Run all validation scripts and tests to confirm no regression. Document the exact count and pass/fail status.

## Exit Criteria

- Landing-gear adapter --> intake packet --> spec --> playback --> diagnosis --> knowledge chain runs end-to-end with no errors.
- Each stage output passes its v1 schema validation (spec, playback, diagnosis, knowledge).
- `tools/validate_landing_gear_full_pipeline.py` exits 0 and emits PASS for all 4 schema cases.
- All 23 shared validation commands pass (no regression).
- `tests/test_landing_gear_full_pipeline_validation.py` passes in both text and JSON output modes.
