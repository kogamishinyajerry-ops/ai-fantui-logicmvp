# P5-02 Plan - Latest-Interaction-Wins Demo Request Arbitration

## Objective

Protect the presenter demo from stale UI state during rapid lever edits, fast condition toggles, and prompt/lever request overlap by ensuring only the newest interaction can rewrite the shared result surface.

## Context

- `P5-01` established GitHub-verifiable HTTP smoke coverage, but the browser shell can still issue overlapping `POST /api/demo` and `POST /api/lever-snapshot` requests.
- The first-screen result area is shared across DemoAnswer and lever snapshot payloads, so an older response arriving late can visually overwrite a newer operator action.
- P5 explicitly targets rapid edits and fast toggles without introducing browser-only approval steps or a second control-truth layer.

## Tasks

1. Add lightweight latest-request-wins arbitration in the browser shell so stale prompt or lever responses are ignored once a newer interaction has started.
2. Clear pending lever debounce work when a prompt submission becomes the newest interaction, so the UI does not replay an older lever intent after the presenter asks a question.
3. Add regression coverage and documentation for this request-arbitration contract, then point the local control plane at `P5-02`.

## Success Criteria

- Fast lever edits, preset clicks, and prompt/lever overlap no longer let an older response repaint the current UI.
- The browser shell still reuses the same `POST /api/demo` and `POST /api/lever-snapshot` contracts without adding new runtime dependencies.
- Regression coverage protects the request-arbitration helpers and the prompt-submit debounce cancellation.
- Local GSD state can treat `P5-02` as the active plan before the next automatic loop step.
