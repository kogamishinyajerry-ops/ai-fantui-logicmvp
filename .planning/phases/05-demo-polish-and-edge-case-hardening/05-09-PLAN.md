# P5-09 Plan - Restore Intuitive TRA Startup And Drag Direction Semantics

## Objective

Correct the cockpit's first-contact TRA interaction so the locked band still follows the `L4` rule, but the slider no longer feels pinned or directionally misleading before unlock.

## Context

- `P5-07` and `P5-08` restored the backend lock rule and the moved VDT live wiring, but the startup interaction still feels wrong in the browser.
- The page booted on a near-threshold preset, so the user landed almost directly on the `-14°` gate with barely any visible leftward travel available.
- The slider labels also fought the numeric range, which made it harder to understand that deeper reverse lives on the left side of the control.
- This is not a controller-truth bug; it is a cockpit interaction semantics bug that makes the correct lock rule feel broken.

## Tasks

1. Move the default startup state away from the `-14°` boundary so the first manual interaction demonstrates the intended free `-14° .. 0°` drag band clearly.
2. Align the TRA rail copy and status text with the actual slider direction so leftward movement corresponds to deeper reverse travel.
3. Preserve the existing presets and `L4` gate rule while extending regression coverage around the new startup semantics.

## Success Criteria

- The page boots from a neutral custom TRA state instead of landing on the `-14°` threshold.
- Before `L4` is satisfied, the user can freely drag within `-14° .. 0°` and receives consistent left/right guidance from the UI copy.
- After `L4` is satisfied, the full `-32° .. 0°` band opens without changing the backend gate rule.
- Static regressions, demo smoke, and the shared validation suite all pass with the corrected startup semantics.
