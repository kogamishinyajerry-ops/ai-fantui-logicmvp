# P6-16 Plan - Bound Prepare-Opus-Review Timeouts

## Objective

Make `prepare-opus-review` as bounded and self-recovering as the newer `run` path, so slow Notion windows stop blocking control-plane refresh and follow-up retries.

## Context

- `P6-14` already made `run` degrade cleanly under slow writeback, and `P6-15` improved how successful maintenance runs preserve the stronger shared validation baseline.
- But the operator loop still has one weak point: `prepare-opus-review` can hang behind slow Notion requests, which forces external process kills and makes it hard to confirm whether the control tower has advanced.
- The safest fix is to bound Notion client socket waits, add a prepare-specific deadline, and fall back to repo docs when the live fetch path times out.

## Tasks

1. Make the Notion client use a configurable, shorter request timeout instead of a hardcoded long wait.
2. Wrap `prepare-opus-review` fetch/write flows in a bounded deadline and degrade to repo-doc snapshots when a timeout occurs.
3. Add focused tests covering timeout fallback behavior so the control-plane refresh path stays deterministic.

## Success Criteria

- `prepare-opus-review` returns promptly during slow Notion windows instead of hanging until an external kill.
- Timeout cases still produce a valid review payload using repo-doc fallback.
- Unit tests cover the new prepare timeout behavior.
