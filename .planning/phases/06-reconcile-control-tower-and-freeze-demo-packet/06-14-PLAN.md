# P6-14 Plan - Add Timeout Fallback To Run Writeback

## Objective

Stop successful `gsd_notion_sync.py run` executions from hanging indefinitely during slow Notion writeback, while still preserving enough live control-plane evidence to keep P6 reconciliation moving.

## Context

- `P6-12` and `P6-13` repaired the dashboard structure and default-plan drift, but the latest verified plan can still lag because a real `run` sometimes stalls during shared Notion database/page writes.
- When that happens, the repo already has enough context to produce a trustworthy fallback snapshot from the just-finished command results.
- The missing piece is a bounded writeback deadline plus a consistent repo-doc sync path that runs on both full success and fallback success, so slow Notion windows stop blocking control-plane advancement.

## Tasks

1. Add a writeback deadline guard around the full Notion `run` persistence path and surface a dedicated timeout/fallback error instead of hanging forever.
2. Reuse the post-run fallback snapshot to keep repo-side coordination docs in sync even when the full Notion write path times out or partially fails.
3. Add focused tests that prove both the fallback path and the full-success path still trigger repo-doc sync from the correct snapshot payload.

## Success Criteria

- A successful `run` returns promptly even if the full Notion writeback path stalls behind slow shared-database/page responses.
- Fallback-success runs still refresh repo-side coordination/freeze docs from the post-run snapshot.
- Unit tests cover both timeout/fallback and full-success repo-doc sync behavior.
