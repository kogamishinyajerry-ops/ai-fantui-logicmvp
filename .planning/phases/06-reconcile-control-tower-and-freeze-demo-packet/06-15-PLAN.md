# P6-15 Plan - Preserve The Stronger Validation Baseline

## Objective

Let control-plane maintenance slices advance the latest verified plan without accidentally downgrading the visible QA baseline from the last full shared validation run.

## Context

- `P6-14` made `run` bounded and resilient under slow Notion writeback, and the control plane can now move forward to the newest P6 slice again.
- But a focused maintenance run that executes only one targeted test still rewrites repo/freeze summaries to `1/1 shared validation checks pass`, which makes the live handoff surfaces look weaker than the real product baseline.
- We need the control plane to distinguish between “latest successful maintenance slice” and “latest stronger shared validation baseline” so current maintenance work can be recorded without erasing richer QA confidence.

## Tasks

1. Teach repo-doc and fallback snapshot recovery to understand the human-readable compact QA summaries that sync already writes.
2. Add a success-evidence selector that preserves the prior richer QA summary when the current run is clearly narrower than the previous shared validation baseline.
3. Cover the new behavior with focused tests around compact-summary parsing, preservation heuristics, and local fallback snapshot construction.

## Success Criteria

- A narrow maintenance run can still become the latest verified plan and latest success run.
- Repo/notion snapshots keep the previous richer QA summary when it represents a stronger shared validation baseline than the current maintenance run.
- Unit tests prove that compact-summary parsing and baseline-preservation behavior work end to end.
