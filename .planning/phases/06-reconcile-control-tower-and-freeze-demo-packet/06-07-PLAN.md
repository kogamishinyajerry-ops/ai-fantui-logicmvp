# P6-07 Plan - Promote Active Snapshot Even When Database Writeback Fails

## Objective

Stop successful control-plane runs from leaving the Notion dashboard, status page, 09C brief, and freeze packet stuck on an older plan when shared database writeback fails under partial integration scope.

## Context

- `P6-06` proved that the active repo docs and Notion pages can still drift apart when a docs-only control-plane slice succeeds in GitHub but the shared Notion database writeback does not fully land.
- We manually repaired that drift by updating the active pages and re-syncing the repo docs, but the loop should be able to self-heal instead of depending on manual page edits.
- The local integration already has enough access to the active pages for a page-only fallback path; the missing piece is promoting the current successful run onto those pages when the full database path fails.

## Tasks

1. Add a fallback snapshot path for successful runs so `handle_run` can still refresh the active pages from page-derived state plus the current plan/run summary when database writeback raises a 404.
2. Let `prepare-opus-review` use the same page fallback when the shared review databases are inaccessible.
3. Add regression tests and rerun the shared validation suite to prove the control plane still passes while the new fallback path stays deterministic.

## Success Criteria

- Successful runs degrade to a page-promotion fallback instead of leaving the active surfaces frozen on the previous verified plan.
- `prepare-opus-review` still works when the token can read/write active pages but cannot query the shared databases.
- The shared validation suite continues to pass with the new fallback logic in place.
