# P7-18 Plan - Add A Visual Second-System Onboarding Readiness Board To The Browser Workbench

## Objective

Make second-system onboarding easier to judge by showing, in the browser workbench, whether a new control-logic packet is complete enough to enter spec build.

## Context

- The current workbench already supports intake assessment, clarification gating, playback, diagnosis, knowledge capture, and browser-side acceptance.
- Users can now compare historical runs visually, but they still need to infer from several cards whether a brand-new control-logic packet is actually complete enough to onboard as a second system.
- The next smallest useful slice is a dedicated readiness board that turns the existing intake-assessment and clarification data into a quick visual answer.

## Tasks

1. Add a browser-side onboarding readiness board that surfaces source-document coverage, component count, logic-node count, scenario count, fault-mode count, and clarification progress.
2. Drive the board from the existing `intake_assessment` and `clarification_brief` payloads so no new backend contract is required.
3. Keep the board aligned with the existing visual acceptance flow by clearly stating what unlocks after completion and what gaps still remain.

## Success Criteria

- Users can tell at a glance whether a new control-logic packet is ready to become a second system in the workbench.
- The readiness board works for both ready and blocked packets and remains consistent with the current bundle truth.
- No extra backend request or parallel rules layer is introduced.
