# P7-54 Plan - Add Second-System Smoke Validation Entry Point

## Objective

Turn the existing custom second-system packet into one repeatable smoke validation entry point, so the repo can explicitly prove a non-reference control system traverses the same intake, bundle, playback, diagnosis, and knowledge chain.

## Context

- The manual Opus 4.6 architecture review called out “second-system smoke validation” as the first real litmus test for whether the workbench abstractions are genuinely reusable.
- The repo already contains `custom_reverse_control_v1.json` plus scattered unit tests around intake, playback, diagnosis, and bundle generation, but there is no single entry point that proves the whole second-system path in one command.
- After `P7-51` through `P7-53` established the spec schema, controller adapter seam, and fault taxonomy, the next strongest evidence is a dedicated second-system smoke report built from the same bundle workflow engineers will actually use.

## Tasks

1. Add a second-system smoke report module that loads the custom packet and proves the full ready bundle chain through one reusable helper.
2. Expose that helper through a stable CLI command for automation and human verification.
3. Add regression coverage proving the smoke report and CLI output confirm the second-system path end to end.

## Success Criteria

- The repo exposes one stable entry point that proves the custom second-system packet completes the intake->bundle->playback->diagnosis->knowledge workflow.
- The CLI can emit a machine-readable second-system smoke report.
- Regression coverage proves the smoke report passes and references the expected second-system scenario and fault mode.
