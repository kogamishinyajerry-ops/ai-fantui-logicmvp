# P7-19 Plan - Add A Visual System Fingerprint Board To The Browser Workbench

## Objective

Make second-system packets easier to understand by showing, in the browser workbench, what kind of system has been loaded before users need to parse bundle detail.

## Context

- `P7-18` already tells users whether a new control-logic packet is ready enough to onboard as a second system.
- That still leaves a separate question unanswered: "What exactly did we ingest?" Users should not need to inspect raw JSON to understand document coverage, control intent, and signal semantics.
- The next smallest useful slice is a dedicated fingerprint board that turns the loaded packet into a visual identity card for the second system.

## Tasks

1. Add a browser-side system fingerprint board that surfaces system ID, control objective, source mode, and source-of-truth note.
2. Show document coverage and concrete signal semantics as visual cards so users can see what was loaded without reading raw JSON.
3. Reuse the existing packet and bundle payloads so the fingerprint board works both before generation and after ready/blocked results arrive.

## Success Criteria

- Users can tell at a glance what kind of second system the current packet represents, not just whether it is ready.
- The fingerprint board works for reference packets, blocked templates, imported local JSON, and completed bundle results.
- No new backend contract or hidden rules layer is introduced.
