# P7-51 Plan - Define System-Agnostic Control System Spec Schema v1

## Objective

Turn the current reference-system workbench spec into an explicit schema-governed JSON contract, so intake output, CLI export, and future second-system adapters can all target one stable control-system spec shape.

## Context

- The manual Opus 4.6 architecture review concluded that the project is over-invested in workbench shell depth and under-invested in the abstraction boundary needed for real system generalization.
- `P7-01` through `P7-50` already built the intake / playback / diagnosis / archive workflow, but the workbench spec payload was still an implicit Python dataclass dump rather than a published JSON contract.
- Before any controller adapter or second-system smoke validation can be credible, the project needs one schema-aware payload shape that is portable outside Python and valid under generic JSON Schema tooling.

## Tasks

1. Add a published `control_system_spec_v1` JSON Schema that documents the reusable control-system workbench payload.
2. Make `system_spec` emit `$schema`, `kind`, `version`, and JSON-safe arrays so generated payloads match the schema contract instead of leaking Python tuples.
3. Extend spec/intake regression coverage so reference spec export and generated intake spec payloads prove the new schema-aware contract.

## Success Criteria

- `docs/json_schema/control_system_spec_v1.schema.json` exists and describes the generated control-system spec payload.
- `workbench_spec_to_dict()` produces a JSON-safe payload with schema metadata and no Python-only tuple structures.
- CLI `spec --format json` and intake-generated specs both surface the same schema/kind/version contract under regression coverage.
