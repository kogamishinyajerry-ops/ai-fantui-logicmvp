# P7-05 Plan - Capture Fault Resolution Knowledge Artifacts

## Objective

Turn a replayed fault diagnosis into a reusable knowledge artifact, so engineer troubleshooting outcomes stop being one-off terminal output and become structured incident, resolution, and optimization records.

## Context

- `P7-04` can now inject a declared `fault_mode` and explain the blocked logic chain through deterministic playback.
- The next missing piece from the product requirement is knowledge accumulation: after engineers diagnose and fix a problem, the system should preserve what happened, what was repaired, how it was validated, and what logic improvements are worth considering next.
- This slice should stay repo-side and spec-driven. It does not need Notion-native persistence yet, but it must emit a machine-readable artifact that is ready to be synced or archived later.

## Tasks

1. Add a knowledge-capture artifact builder that wraps a diagnosis report with incident, resolution, and optimization records.
2. Expose that artifact through the CLI with enough inputs to represent both “diagnosed” and “resolved” troubleshooting outcomes.
3. Add focused tests proving the current intake fixture can produce a resolved knowledge artifact with evidence links and optimization suggestions.

## Success Criteria

- A ready intake packet can produce a structured knowledge artifact from `scenario_id + fault_mode_id` plus engineer resolution details.
- The artifact clearly separates incident facts, repair/validation details, and post-repair optimization advice.
- The output remains machine-readable and suitable for later sync into docs, Notion, or other storage.
- No cockpit demo endpoint or presenter-facing workflow changes are introduced.
