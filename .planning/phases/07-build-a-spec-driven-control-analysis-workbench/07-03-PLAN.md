# P7-03 Plan - Compile Intake Scenarios Into Monitor-vs-Time Playback Traces

## Objective

Turn a ready mixed-doc intake packet into an executable monitor-vs-time playback trace, so engineer-supplied process descriptions stop being passive documents and become reproducible scenario evidence.

## Context

- `P7-01` created the reusable spec foundation, and `P7-02` created the mixed-doc intake packet plus readiness assessment.
- The next missing step is the first strict playback engine: given a ready intake packet and a named acceptance scenario, the system should compile signal timelines, evaluate simple logic nodes, and produce a machine-readable trace suitable for later charting and diagnosis.
- This slice should stay focused on deterministic scenario playback. Fault trees, diagnosis narratives, and optimization suggestions can layer on top after the playback evidence exists.

## Tasks

1. Extend acceptance scenarios with explicit steady-signal support so rate-driven ramps and held inputs can both be expressed cleanly.
2. Add a generic scenario playback engine that samples scenario transitions over time, evaluates logic nodes, and emits monitored signal + logic series.
3. Expose playback through the CLI and add tests proving an intake packet can become a reproducible monitor-vs-time trace.

## Success Criteria

- A ready intake packet can be replayed into sampled time-series data without touching the UI.
- Binary steady inputs and analog ramps can coexist in one scenario description.
- Derived logic-node states and downstream commands appear in the playback output when their conditions are satisfied.
- The new playback feature introduces no changes to the existing cockpit demo endpoints or presenter flow.
