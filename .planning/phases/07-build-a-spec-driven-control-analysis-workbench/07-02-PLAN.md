# P7-02 Plan - Add Mixed-Document Intake And Custom Signal Semantics Assessment

## Objective

Create a concrete intake layer for new control systems so mixed documents/PDF references, custom signal semantics, and unanswered onboarding questions can be normalized before scenario playback or fault diagnosis begins.

## Context

- The user clarified that near-term source material will be mixed docs/PDF rather than clean YAML-only inputs.
- The user also clarified that signals like `VDT` must be interpreted per system, with fully custom units and semantics, not inherited from the current reference chain.
- That means the next safe automation step is not direct PDF parsing. It is a canonical intake packet plus validation/assessment logic that can tell us when a new system packet is complete enough to build a workbench spec.
- Without this layer, future system onboarding would drift into ad hoc assumptions, which directly violates the requirement to keep asking until the details are actually understood.

## Tasks

1. Add a mixed-document intake packet model that records source-document references, custom component semantics, scenario timelines, fault modes, and clarification answers.
2. Add normalization/assessment logic that flags blocking gaps, especially missing clarification answers and malformed signal semantics.
3. Expose that layer through a small CLI surface so future automation can print a starter template and assess incoming packets without touching the UI first.

## Success Criteria

- A new system can be represented as a single JSON packet even when its source evidence spans PDFs and markdown notes.
- The intake assessment explicitly blocks spec generation when required clarification answers are missing.
- Custom signals carry system-defined units and state shapes instead of inheriting assumptions from the reference thrust-reverser chain.
- The CLI can export a starter intake template and emit a machine-readable intake assessment report.
