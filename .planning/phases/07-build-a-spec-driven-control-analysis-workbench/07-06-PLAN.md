# P7-06 Plan - Emit Clarification Follow-Up Briefs For New-System Onboarding

## Objective

Turn intake gating into an engineer-facing clarification brief, so future-system onboarding does not stop at a vague `not ready` result and instead produces the exact follow-up questions and unlock path needed before replay or diagnosis can proceed.

## Context

- `P7-02` already introduced mixed-doc intake packets plus readiness assessment, and `P7-03` through `P7-05` now cover playback, fault diagnosis, and knowledge capture once a packet is ready.
- The remaining roadmap gap is operational ambiguity handling: when a new control system packet is incomplete, the tool must keep asking rather than silently guessing.
- The safest next slice is not another simulation feature. It is an explicit clarification artifact that engineers can act on immediately.

## Tasks

1. Add a structured clarification-brief builder on top of intake assessment, including question status, rationale, gating statement, and next actions.
2. Expose that brief through the existing `well_harness intake` CLI so engineers can export either the standard readiness report or a dedicated follow-up brief.
3. Add focused tests covering both ready and blocked intake packets, plus the CLI JSON path for the new brief.

## Success Criteria

- An intake packet with unanswered questions can produce a machine-readable follow-up brief instead of only a generic readiness failure.
- The brief explicitly states what is missing, why it matters, and what capability unlocks after the answers are supplied.
- A ready packet reports that the clarification gate is clear and can proceed into spec build / playback / diagnosis / knowledge capture.
- No cockpit UI, controller truth, or demo API contract changes are introduced.
