# P7-25 Plan - Protect Browser Packet Drafts Before Overwrite

## Objective

Make browser-side packet editing safer by turning manual JSON drafts into first-class saved revisions before later UI actions overwrite them.

## Context

- `P7-23` added packet revision history and restore.
- `P7-24` added a compare strip for historical packet revisions versus the latest packet.
- Engineers can still lose valid manual JSON edits if they switch samples, restore another packet version, or apply browser-side writebacks before capturing the current draft.

## Tasks

1. Add a visible packet-draft status bar and a manual “save current packet draft” action inside the browser workbench.
2. Auto-save valid unsaved packet drafts before actions that overwrite the current editor payload, such as sample loads, packet restores, and browser-side packet writebacks.
3. Keep the draft surface honest by disabling save when the JSON is invalid or already synced to the selected saved revision.

## Success Criteria

- Engineers can explicitly save a packet draft even before rerunning the workflow.
- Valid in-browser packet edits survive later overwrite actions without forcing users to remember to run first.
- Invalid JSON is never silently treated as a saved packet revision.
