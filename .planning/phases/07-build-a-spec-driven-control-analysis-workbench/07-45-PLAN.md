# P7-45 Plan - Make Archive Manifest Paths Portable

## Objective

Make saved workbench archives stay valid after the archive directory is moved by switching manifest path references from archive-local absolute paths to archive-relative paths.

## Context

- `P7-34` introduced `archive_manifest.json`, and later slices added validation, CLI inspection, self-check metadata, schema coverage, and README/schema references.
- The current manifest still stores archive-local path references in a way that effectively ties validation to the archive's original filesystem location.
- For handoff and restore, a moved archive package should still validate from its new location.

## Tasks

1. Store archive-local manifest file references and restore targets as relative paths.
2. Resolve relative `archive_dir` / file paths against the manifest file location during validation and loading, while keeping legacy absolute-path manifests valid in place.
3. Cover moved-archive validation and updated manifest/demo assertions with regression tests.

## Success Criteria

- Newly generated manifests store archive-local relative paths and `archive_dir` as a portable relative marker.
- `load_workbench_archive_manifest()` and the CLI validator succeed after a generated archive directory is moved.
- Existing same-machine legacy absolute-path manifests remain loadable.
