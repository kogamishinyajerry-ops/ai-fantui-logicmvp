# P7-09 Plan - Add A Browser Acceptance UI For Workbench Bundles

## Objective

Expose the new workbench bundle + archive workflow through a lightweight browser surface, so users can validate ready and blocked intake packets without stitching together CLI commands by hand.

## Context

- `P7-07` already composes intake, clarification, playback, diagnosis, and knowledge capture into a single engineer-facing bundle.
- `P7-08` already archives that bundle into a durable JSON + Markdown package, but the only current access path is still shell-first.
- The next smallest slice is not new reasoning logic. It is an acceptance shell: reuse the existing bundle/archive truth through `demo_server`, preload a known-good reference packet plus a blank template, and let the user exercise the workbench from the browser.

## Tasks

1. Extend `well_harness.demo_server` with bootstrap and bundle endpoints that expose the existing workbench bundle/archive helpers without introducing a second control-truth layer.
2. Add a dedicated `workbench.html` acceptance page plus browser-side JS/CSS to load a reference packet, load a blank template, import local JSON, and render ready-versus-blocked bundle results.
3. Add focused tests for the new page, bootstrap payload, and archive-producing bundle POST flow so the acceptance surface is regression-protected.

## Success Criteria

- `/workbench.html` provides a clear acceptance path for generating workbench bundles from the browser, including optional archive package output.
- The UI can exercise both a ready reference packet and a blocked template packet while reusing the existing intake / clarification / playback / diagnosis / knowledge / archive logic.
- `demo_server` remains a thin presentation layer over the existing workbench helpers and does not replace `controller.py` or invent a parallel rule engine.
