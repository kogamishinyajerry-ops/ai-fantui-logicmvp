# P7-53 Plan - Publish Reusable Fault Taxonomy Contract

## Objective

Turn the project's supported fault kinds into an explicit reusable taxonomy contract, so intake packets, system specs, and diagnosis logic all agree on the same `fault_kind` vocabulary.

## Context

- The manual Opus 4.6 architecture review called out the lack of a general fault-model abstraction as a major blocker to true system generalization.
- The repo already uses several implicit fault kinds such as `bias_low`, `stuck_low`, `latched_no_unlock`, and `command_path_failure`, but they were only enforced by scattered string comparisons.
- After `P7-51` established a schema-aware spec contract and `P7-52` introduced the controller truth adapter seam, the next smallest core abstraction is a published taxonomy for fault semantics.

## Tasks

1. Add a reusable fault-taxonomy module plus a published JSON Schema for the supported fault-kind vocabulary.
2. Make intake parsing and runtime diagnosis validate `fault_kind` against that taxonomy instead of accepting arbitrary strings.
3. Tighten the control-system spec schema and regression coverage so generated specs and intake packets stay inside the published taxonomy contract.

## Success Criteria

- The repo contains a documented, machine-readable fault taxonomy contract for supported fault kinds.
- Intake parsing rejects unknown `fault_kind` values with a helpful validation error.
- Existing reference specs, second-system packets, and diagnosis tests continue to pass under the stricter taxonomy boundary.
