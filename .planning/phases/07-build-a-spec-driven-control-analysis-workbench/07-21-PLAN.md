# P7-21 Plan - Add An In-Browser Clarification Refill Workspace

## Objective

Turn blocked clarification runs into a continuous browser workflow by letting engineers answer clarification items, write those answers back into the intake packet, and rerun without leaving the workbench surface.

## Context

- `P7-20` already shows which clarification items, schema blockers, and unlocks are currently relevant.
- The current workbench still requires users to drop back to raw packet JSON if they want to answer a clarification and try again.
- The next step should keep reusing the existing intake packet and bundle payload truth instead of introducing a frontend-only rules layer.

## Tasks

1. Add a visual clarification refill workspace that turns pending `follow_up_items` into editable answer cards in the browser workbench.
2. Let the workspace write answers back into `packet_payload.clarification_answers` and optionally rerun the same bundle flow from the browser.
3. Keep the refill workspace aligned with preparation, running, blocked, ready, and failed states so it never pretends to know more than the backend payload says.

## Success Criteria

- A blocked clarification bundle exposes its unanswered clarification items as editable cards in the browser.
- Users can write answers back into the current intake packet without manually editing raw JSON.
- Users can immediately rerun after writing answers, while the frontend continues to reuse existing payload truth instead of inventing a second rules engine.
