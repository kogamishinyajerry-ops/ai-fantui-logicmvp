# P7-14 Plan - Make Recent Acceptance Results Reopenable In The Browser Workbench

## Objective

Turn the recent acceptance history board into a real replay surface, so users can click an earlier pass, block, or failure card and restore that exact result onto the main visual board.

## Context

- `P7-13` preserved the latest few workbench outcomes, which solved the “previous result disappeared” problem.
- The remaining friction is that those cards still behave like static receipts; users can see that a run happened, but they cannot bring that specific state back into focus without rerunning the workflow.
- The next smallest usability slice is to let the browser restore a stored result from history while reusing the same existing bundle / failure rendering surfaces.

## Tasks

1. Store enough structured result data inside each recent-run history entry to re-render the main visual board later.
2. Make each history card clickable, with clear selected-state styling and copy that explains it can reopen a prior result.
3. Reuse the existing bundle and failure rendering paths for history replay, without creating a separate UI-only result format.

## Success Criteria

- A user can click a recent history card and immediately see that earlier result restored onto the main visual board.
- Ready, blocked, archived, and request-failure entries all remain compatible with the same replay surface.
- History replay does not create duplicate history entries or desynchronize the main verdict board from the detailed result panels.
