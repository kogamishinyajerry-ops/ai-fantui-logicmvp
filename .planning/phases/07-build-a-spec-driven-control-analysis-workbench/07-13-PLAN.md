# P7-13 Plan - Add A Recent Acceptance Run History To The Browser Workbench

## Objective

Keep consecutive browser workbench acceptance runs visible, so users can compare pass, block, archive, and failure outcomes without losing context to the most recent result repaint.

## Context

- `P7-12` already made the main workbench acceptance flows one-click and ensured the latest click wins.
- That improved speed, but it also made comparison harder: once a new preset finishes, the previous result disappears from the visible acceptance surface.
- The next smallest usability slice is to preserve a short recent-run history on the same page, while staying anchored to the existing bundle-generation truth instead of inventing a second reporting layer.

## Tasks

1. Add a recent acceptance history board to the browser workbench shell so the page reserves visible space for consecutive run outcomes.
2. Record ready, blocked, archived, and request-failure outcomes from the existing frontend bundle flow and render them as compact cards with labels, summaries, and timestamps.
3. Keep the feature aligned with the current visual acceptance direction by making it additive to the existing verdict board, not a replacement for it.

## Success Criteria

- Users can run multiple presets in sequence and still see the recent pass/block/archive trail without reading raw JSON.
- The history cards are fed by the existing bundle request lifecycle and do not introduce UI-only verdict logic.
- Request failures are visible in the same recent-run surface, so debugging still stays inside the visual acceptance workflow.
