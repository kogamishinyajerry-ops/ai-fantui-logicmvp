# P7-23 Plan - Add A Packet Revision History Board To The Browser Workbench

## Objective

Make browser-side packet editing safer and more explainable by keeping a visible packet revision history that can restore earlier intake packet versions before rerunning the workbench flow.

## Context

- `P7-21` lets engineers write clarification answers back into the packet from the browser.
- `P7-22` lets engineers apply backend-approved safe schema repairs from the same surface.
- Now that the browser can mutate the intake packet directly, engineers need a trustworthy way to see which packet version is current and to recover an older version without reconstructing it from memory.

## Tasks

1. Add a packet revision history board to the workbench input surface.
2. Capture packet revisions for sample loads, local JSON imports, browser-side clarification writebacks, safe schema repairs, and run-time manual JSON edits before bundle execution.
3. Let users restore an older packet revision back into the current input surface before rerunning the flow.

## Success Criteria

- The browser workbench shows a visible history of packet revisions instead of silently overwriting the current packet.
- Engineers can restore a previous packet version with one click.
- Restored packet versions reset the main board back into a “rerun this version” preparation state instead of pretending old output still matches the current input.
