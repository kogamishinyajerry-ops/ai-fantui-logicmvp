# P7-22 Plan - Add A Safe Schema Repair Workspace To The Browser Workbench

## Objective

Make blocked schema issues more actionable by exposing backend-approved safe repair suggestions in the browser workbench and letting users apply those repairs before immediately rerunning the same bundle flow.

## Context

- `P7-21` already lets engineers answer clarification items inside the browser workbench and rerun without dropping to raw JSON first.
- The next main gap is schema blockers: users can see them, but safe template-like fixes still require hand-editing packet JSON.
- Any schema repair entrypoint must stay grounded in backend truth instead of letting the frontend invent patch logic.

## Tasks

1. Extend intake assessment with structured repair suggestions that distinguish safe autofix items from manual-only schema blockers.
2. Add a demo-server repair endpoint that applies only backend-approved safe schema fixes to the current packet payload.
3. Expose those safe repair suggestions in the browser workbench and let users apply all safe fixes and rerun from the same acceptance surface.

## Success Criteria

- Blocked packets show which schema blockers can be safely auto-repaired and which still require manual editing.
- The browser can apply safe schema fixes without introducing a second frontend rules layer.
- After applying safe fixes, the user can immediately rerun the bundle flow from the same page.
