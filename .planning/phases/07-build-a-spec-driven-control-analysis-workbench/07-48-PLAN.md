# P7-48 Plan - Connect Archive Restore In Workbench UI

## Objective

Make the new archive-restore backend reachable from the browser workbench so engineers can reopen a saved archive package without leaving the acceptance surface.

## Context

- `P7-47` added a stable `/api/workbench/archive-restore` entrypoint that reopens moved archive packages.
- The browser workbench still lacked a UI affordance for that flow, which forced archive restore back into raw API calls despite the rest of the workbench already living in the browser surface.
- The workbench already knows how to restore browser workspace snapshots, so the safest next slice is to reuse that same restore path after fetching archive restore payloads from the backend.

## Tasks

1. Add a browser input and action to submit an archive manifest path or archive directory to the restore API.
2. Reuse the existing workspace-snapshot restore flow when archive restore returns browser workspace state, and fall back to bundle-summary restore when only archived bundle metadata is available.
3. Cover the new UI hooks and moved-archive directory restore contract in regression tests.

## Success Criteria

- The workbench UI exposes a visible “从 Archive 恢复工作区” affordance.
- Users can paste either `archive_manifest.json` or the archive directory path and successfully reopen archive-backed workspace state through the browser workbench.
- Existing bundle generation, archive export, and restore backend behavior remain green.
