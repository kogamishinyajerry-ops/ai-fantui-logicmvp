# P7-27 Plan - Persist Browser Acceptance Result History Across Refresh

## Objective

Keep the browser workbench's result-side context alive across refreshes by restoring recent acceptance results, replay/latest view state, and history sequencing from browser storage.

## Context

- `P7-26` already keeps the packet-side workspace alive across refreshes.
- Engineers can still lose the result-side context after a refresh, even when the current packet, packet history, and inputs come back correctly.
- The workbench already has replay/latest result semantics, so browser restore should reopen those results honestly instead of forcing users to rerun just to recover context.

## Tasks

1. Persist recent acceptance result history plus the current replay/latest view state into the existing browser workspace snapshot.
2. Restore that result-side context after bootstrap so refreshes reopen the latest or replayed result instead of always dropping to a preparation-only board.
3. Continue history ID sequences after restore so newly created run-history or packet-history entries do not collide with restored browser history.

## Success Criteria

- Refreshing the workbench restores recent acceptance results together with the current replay/latest result mode.
- Users no longer need to rerun immediately after refresh just to recover the latest or replayed result context.
- Restored browser history continues to accept new history entries without ID collisions.
