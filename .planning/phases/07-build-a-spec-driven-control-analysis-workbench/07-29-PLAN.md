# P7-29 Plan - Harden Notion Control-Plane Development Rules

## Objective

Make the Notion-controlled automation loop publish the current development architecture and anti-drift execution rules explicitly, so new Codex sessions do not mistake local progress for a fully synced control plane.

## Context

- The repo already has a working `gsd_notion_sync.py` writeback path, and live `run` / `prepare-opus-review` are healthy again.
- The dashboard, status page, and repo-side handoff docs already auto-sync current plan/run/gate evidence, but they do not yet state the stricter completion contract that this workflow now depends on.
- That omission makes it too easy for a future Codex session to stop after local code and tests, without noticing that live Notion writeback and re-validation are also part of the definition of done.

## Tasks

1. Add a shared “development architecture and execution rules” section to the auto-synced Notion/status/repo handoff renderers in `tools/gsd_notion_sync.py`.
2. Update the synced dashboard rules so the standard loop explicitly requires `gsd_notion_sync.py run`, `prepare-opus-review`, and treating writeback degradation as a control-plane blocker.
3. Extend the control-plane tests so CI fails if those anti-drift rules disappear from the synced outputs.

## Success Criteria

- Auto-synced control-plane surfaces explicitly say GitHub/repo is implementation truth, Notion is control plane, and `controller.py` remains the single control truth.
- The synced rules explicitly require live Notion writeback plus `prepare-opus-review` re-check before a slice is treated as complete.
- A regression that removes those rules from the synced outputs fails local tests.
