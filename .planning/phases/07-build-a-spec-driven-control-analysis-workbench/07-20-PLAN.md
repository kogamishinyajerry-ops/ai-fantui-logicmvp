# P7-20 Plan - Add A Visual Onboarding Action Board To The Browser Workbench

## Objective

Make blocked second-system onboarding runs actionable by showing, in the browser workbench, which clarification answers and schema fixes are still needed before the next unlock.

## Context

- `P7-18` tells users whether a packet is ready enough to onboard as a second system.
- `P7-19` tells users what kind of system they loaded by showing document coverage, control objective, source-of-truth note, and signal semantics.
- Users still need a clearer answer to "what do I fix next?" because the existing next-actions area is still mostly text-first.

## Tasks

1. Add a browser-side onboarding action board that separates pending clarification items, schema blockers, and unlocks into readable visual columns.
2. Drive the board only from the existing `clarification_brief`, `blocking_reasons`, and `unlocks_after_completion` payloads so the frontend does not invent a second rules layer.
3. Keep the board useful for ready, blocked, preparation, running, and failed states so the user always sees what the current action surface means.

## Success Criteria

- Blocked packets clearly show which clarification answers and schema fixes are still outstanding.
- Ready packets clearly show that the path is open and what capabilities are now unlocked.
- No backend contract change or hidden rule engine is introduced.
