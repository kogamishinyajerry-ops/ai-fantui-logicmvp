# P7-11 Plan - Turn The Browser Workbench Into A Visual Acceptance Board

## Objective

Make the browser workbench readable as a visual acceptance surface, so non-technical users can tell whether a bundle passed, where it is blocked, and whether it has been archived without having to read raw JSON or code-heavy output.

## Context

- `P7-09` and `P7-10` already made the workbench accessible from the browser and exposed the full knowledge-capture contract.
- The remaining usability gap is presentation: the page still reads like a structured debugging console, which is fine for engineers but not for plain-language acceptance.
- The next smallest slice is UI-first: keep the backend payload intact, add a one-glance verdict board and stage strip, demote raw JSON to opt-in debug drawers, and fix any newly exposed repeated-run stability issues discovered while exercising the live archive flow.

## Tasks

1. Add a visual acceptance board that summarizes pass/block/archive state and shows where the current run is in the intake -> clarification -> playback -> diagnosis -> knowledge -> archive chain.
2. Reduce the visual weight of raw JSON and advanced packet fields by moving them into explicit dev/debug drawers, while keeping the one-click reference/template acceptance path front and center.
3. Fix and regression-test same-second archive name collisions so repeated archive clicks do not crash the browser-backed workbench flow.

## Success Criteria

- A user can open `/workbench.html`, load a sample, click generate, and understand the result primarily from the visual board and stage strip.
- Raw JSON and other code-shaped surfaces remain available for debugging but are no longer the default acceptance path.
- Repeated archive requests within the same second create unique archive directories instead of raising `FileExistsError`.
