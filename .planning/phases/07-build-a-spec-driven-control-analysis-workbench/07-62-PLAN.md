# P7-62 Plan - Publish Reusable Workbench Bundle Contract

## Objective

Turn the engineer-facing workbench bundle into a formal schema-aware wrapper, so the combined intake, clarification, playback, diagnosis, and knowledge handoff has the same machine-readable contract boundary as the core artifacts it contains.

## Context

- `P7-56` through `P7-61` formalized the core spec -> playback -> diagnosis -> knowledge chain and promoted its validation into the shared suite.
- But `WorkbenchBundle.to_dict()` still behaved like an implicit dataclass dump, even though it is the main handoff shape used by CLI, archive packages, API responses, and browser workbench flows.
- The next highest-value convergence step is to formalize the bundle wrapper itself while preserving the existing nested artifact contracts and consumer-visible fields.

## Tasks

1. Add explicit schema metadata and contract-aware serialization to `workbench_bundle`.
2. Publish `docs/json_schema/workbench_bundle_v1.schema.json` describing ready and blocked bundle payloads with nested schema-aware artifact envelopes.
3. Add focused regression coverage proving ready and blocked bundles carry the new wrapper and validate against the published schema when `jsonschema` is available.

## Success Criteria

- Workbench bundle JSON exposes a stable `$schema` / `kind` / `version` contract.
- `docs/json_schema/workbench_bundle_v1.schema.json` documents the combined bundle payload shape.
- Regression coverage proves ready and blocked bundle payloads validate against the published schema.
