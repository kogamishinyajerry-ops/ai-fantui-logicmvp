# P7-10 Plan - Surface Diagnosis And Knowledge Details In The Browser Workbench

## Objective

Upgrade the browser acceptance surface from a thin bundle trigger into a readable engineer workbench, so playback, diagnosis, knowledge, and optimization details can be validated directly from the UI instead of only through raw JSON.

## Context

- `P7-09` already lets users generate ready and blocked workbench bundles from `/workbench.html`.
- The remaining gap is visibility: the backend and archive already carry rich diagnosis and optimization data, but the browser UI only exposes a small top-line summary plus raw JSON.
- The next smallest slice is presentation parity. Reuse the existing bundle payload, expose the remaining knowledge-capture inputs, and render the most important component summaries in structured cards.

## Tasks

1. Add the missing browser-side inputs for suggested logic change, reliability gain hypothesis, and guardrail note so the UI can exercise the full knowledge-artifact contract.
2. Render playback, diagnosis, knowledge, and optimization summaries as structured output cards on `/workbench.html`, with readable fallback states for blocked bundles.
3. Extend the focused demo-server tests so the HTML shell and `/api/workbench/bundle` route are regression-protected for the new optimization-field round-trip.

## Success Criteria

- The browser workbench can submit and display the full knowledge-capture field set without dropping optimization inputs.
- Ready bundles show structured playback / diagnosis / knowledge / optimization summaries directly in the UI, while blocked bundles remain readable and honest about missing downstream artifacts.
- The acceptance surface stays a thin view over the existing workbench bundle contract and does not introduce a parallel rule engine or UI-only truth layer.
