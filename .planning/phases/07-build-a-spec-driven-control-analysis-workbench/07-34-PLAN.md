# P7-34 Plan - Add Archive Manifest For Workspace Artifacts

## Objective

Give each saved workbench archive package a single machine-readable manifest so later restore, sync, and audit tooling no longer has to infer the archive structure by scanning filenames.

## Context

- `P7-32` and `P7-33` already made archive packages preserve browser handoff context and the full browser workspace snapshot.
- Those artifacts are useful, but the archive still lacks a single authoritative file map that tells downstream tooling what exists, what kind of bundle was saved, and which files matter for browser recovery.
- The archive generator already knows every path it wrote, so it can emit a manifest cheaply and expose it through the API and browser archive file list.

## Tasks

1. Add a backend-owned `archive_manifest.json` artifact that records bundle identity, archive metadata, written file paths, and restore targets.
2. Point archive README text at that manifest so humans and tools converge on the same file map.
3. Expose the manifest through API/static tests and the browser archive file list.

## Success Criteria

- Archive packages include a machine-readable `archive_manifest.json`.
- The manifest records bundle metadata, file paths, and restore targets for workspace handoff/snapshot recovery.
- Browser/API/archive tests cover the manifest without regressing existing archive behavior.
