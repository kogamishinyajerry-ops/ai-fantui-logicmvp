# P7-01 Plan - Introduce A Reusable Control-System Spec Foundation

## Objective

Create the first reusable control-system specification layer so future work can simulate engineer-supplied process docs, inject faults, produce diagnosis artifacts, and onboard new systems without guessing through ambiguity.

## Context

- The project started as a deterministic thrust-reverser cockpit demo, but the new product requirement is broader: it must support strict engineer-facing acceptance playback, fault injection, knowledge accumulation, and rapid migration to new systems.
- The current codebase already contains one confirmed control chain in `src/well_harness/controller.py`; this should become the first reference system, not be replaced by a second hidden rule engine.
- The biggest near-term risk is not UI polish. It is architectural drift: if scenario playback, diagnosis, and future-system onboarding each invent their own data model, the system will become impossible to verify.
- The first step therefore needs to be a shared spec layer that can describe components, logic nodes, monitored signals, acceptance scenarios, fault modes, and mandatory clarification questions.

## Tasks

1. Add a reusable Python data model for a control-system workbench spec, including component/state definitions, logic nodes, acceptance scenarios, fault modes, and onboarding clarification questions.
2. Capture the current thrust-reverser deploy chain as the first reference spec so the generic layer is grounded in real code truth instead of abstract theory.
3. Add focused tests that prove the reference spec is complete enough to drive later phases for strict acceptance playback, fault diagnosis, and future-system onboarding.

## Success Criteria

- There is a single reusable spec object that future scenario playback, fault injection, and knowledge capture work can all build on.
- The reference spec clearly distinguishes binary switches/commands, fixed-voltage components, analog monitored quantities, logic nodes, and scenario timelines.
- The current reference system includes explicit fault-analysis hooks and explicit clarification questions for future unknown systems.
- The work introduces no changes to the current demo endpoints, controller outputs, or presenter UI behavior.
