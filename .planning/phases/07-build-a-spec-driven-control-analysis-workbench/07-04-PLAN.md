# P7-04 Plan - Inject Fault Modes And Emit Chain Diagnosis Reports

## Objective

Turn a declared intake `fault_mode` into a reproducible diagnosis artifact, so the workbench can move from passive playback into deterministic fault injection and chain-aware troubleshooting.

## Context

- `P7-03` made acceptance scenarios executable as monitor-vs-time playback traces.
- The next missing capability is fault-facing, not UI-facing: engineers need to inject a declared fault, compare baseline vs faulted playback, and get a readable reasoning report that follows the logic chain.
- This slice should stay deterministic and spec-driven. It does not need full repair workflow capture yet, but it must prove the system can replay a fault and explain why a downstream command no longer appears.

## Tasks

1. Add a reusable fault-diagnosis engine that injects supported fault kinds into scenario playback and compares baseline vs faulted traces.
2. Produce a structured diagnosis report with affected signals, blocked logic nodes, scope observations, reasoning steps, and optimization guidance.
3. Expose the diagnosis path through the CLI and add focused tests for the current mixed-doc intake fixture.

## Success Criteria

- A ready intake packet plus `scenario_id` and `fault_mode_id` can produce a machine-readable diagnosis report without touching the demo UI.
- The diagnosis output makes the causal chain explicit enough to guide troubleshooting, not just say "the scenario failed".
- The current fixture proves at least one injected fault can block an expected downstream command and surface the relevant logic blocker.
- The work introduces no changes to existing cockpit demo endpoints or presenter-facing behavior.
