# P7-07 Plan - Bundle The Workbench Chain Into A Single Engineer Artifact

## Objective

Turn the now-separate intake, clarification, playback, fault diagnosis, and knowledge-capture primitives into one engineer-facing workbench bundle so onboarding and troubleshooting no longer require hopping across multiple commands to reconstruct a single story.

## Context

- `P7-02` through `P7-06` already established the workbench chain: mixed-doc intake, clarification gating, deterministic playback, fault diagnosis, and knowledge capture.
- The main remaining workflow gap is orchestration. Today the pieces exist, but engineers still have to run them one by one and manually assemble the combined output.
- The safest next slice is composition, not a new logic model: add a single bundle artifact and CLI surface that reuse the existing deterministic primitives without changing controller truth, demo APIs, or the intake schema itself.

## Tasks

1. Add a reusable workbench-bundle builder that always carries intake assessment + clarification state, and conditionally stitches in playback, fault diagnosis, and knowledge capture when the packet is ready.
2. Expose the bundle through `well_harness` as a single CLI command that can auto-resolve the only scenario / fault mode when the packet is unambiguous.
3. Add focused tests covering both blocked-packet follow-up output and the ready-packet full-bundle path.

## Success Criteria

- A blocked intake packet can emit a single clarification-follow-up bundle instead of forcing engineers to inspect multiple separate reports.
- A ready packet with one scenario and one fault mode can emit a unified machine-readable bundle containing intake, clarification, playback, diagnosis, and knowledge sections.
- The new bundle is compositional only: no cockpit UI contract, controller truth, or hidden second rule engine is introduced.
